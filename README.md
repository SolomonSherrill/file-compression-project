NOTE: I used AI to create the web version of my LZW encoder/decoder, and I used it to help me learn about how to work with byte data (headers for lzw, etc), in both my huffman and lzw files. All logic is my own.

This project taught me quite a bit about how file compression works, and especially when compression will help reduce a files size.

LZW: Building this compression algorithm was not insanely hard. My first step was researching different compression algorithms, and I decided to use LZW. From there, I looked up how LZW worked, and found a basic pipeline (not direct code) on its process, then implemented it myself in Python. Following this, I learned that LZW works as a universal compression algorithm, so I slightly tweaked the logic for a method that would assign codes based on sequences of binary rather sequences of ASCII. The decoder was slightly harder to build, as I did find it conceptually harder to understand, but after watching a video on it it wasn't wildly hard to implement. 

Huffman: This algorithm was substantially harder to build than LZW. To assist in creating the binary tree, I created a class node that can store a character and its associated frequency in a text. This later made it much easier to create binary trees using a heap. The first step was counting the frequency of each character, and following this, I created a heap of each character with its associated frequency in the scanned text. Utilizing a greedy algorithm with the heap to create a huffman binary tree, I select the two nodes with the min frequency in the heap using heappop, and assigning their parent node as the combined frequency of the two. By iteratively doing this, a Huffman binary tree is built from the ground up. This took me quite a bit of time to understand conceptually, but it allows for the huffman binary tree to be build substantially faster than by using other methods such as sorting a list. After this tree is built, characters are assigned codes based on their position on the tree. I later learned that codes generated by compressing files using LZW could be compressed again with huffman compression, and this allows for substantially higher compression.

LZW + Huffman: Requires little documentation. Essentially, the codes outputted by lzw are fed into huffman encoding, which reduces size taken up by frequent codes.

My own algorithm (green eggs): I am quite proud of this. NOTE: though this algorithm assigns tokens binary strings, the strings being frequency based does not impact compression size, as I decided to save bits as fixed length to avoid hassles with variable bit length saving. My algorithm first reads a file and assigns each word string (separated by spaces) a frequency, which is later turned into a wordlist which is simply a list of words, which is sorted. Newlines and spaces are also handled as their own strings. After, each string is assigned a binary code based on frequency (reference above note), where the length of each code is dependent on the total number of words, so the length of each binary code is variable to save additional space. After this, an encoded string is saved containing only binary referencing word strings in the original text. After this, it is saved as binary, and an additional json is saved containing each word string with its associated binary code, and the length of each code for decoding. To decode, the binary string is turned into a bitstring. Then the json is read returning the code length and the reversed dict (bitstring: word) of the word codes. From there, decoding is simple and the code just iterates through the bitstring and converts the binary into original word, where newlines are converted to \n

Graphing: The graphing of the effectiveness of the algorithms is quite simple and requires little documentation. Essentially, all algorithm classes are imported, and from there, a file is selected. Next, the file size of the output of all compression algorithms is compared to determine compression %. Graphing is done with matplotlib.
